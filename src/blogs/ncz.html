我最近在忙于一个项目, 本来使用Espress( ESLint的Javascript解析器 ), 后来改使用Acorn(也是一个Javascript解析器). 然而, 这却让我遇到一个很有趣的关于 ` Object.defineProperty()` 的性能问题.

而且在V8引擎中(包括Chrome 与 Node.js), 似乎只要是调用 `Object.defineProperty()`就会明显对性能产生不好的影响. 仔细研究后我得到了一些很有意思的结果.


问题 : 

我第一次注意到这个问题是在做ESLint性能测试的时候, 我发现当我使用 Acorn-powered Espree 时, 执行时间会增加500ms. 而当我使用最新版本的Espree (v2.2.5), 执行时间只有2500ms. 当我再切换到Acorn-powered Espre, 执行时间又增加到3000ms. 500ms看起来很少, 但对于ESlint 用户来说这是非常巨大的变化了, 所以我决定找出来什么原因导致执行时间增加这么多.


调查:

在Chrome的profiler面板, 我使用ESLint的性能分析命令(npm run profile)去测试ESlint. 然后发现了下面的情况 :




Soga, 如图可以明显看到`esprimaFinishNode()`函数的执行时间占总执行时间的33%. 这个函数的功能是 从Acorn添加AST nodes, 这是为了让它们看起来更像Esprima AST nodes. (读者不必关心这个函数的作用). 我瞬间就意识到唯一的可能操作就是Object.defineProperty().

Acorn 为每个 AST node 添加了非标准的 start 与  end 属性, 这是为了追踪它们的位置. 而且这俩个属性绝对不可以被删除, 因为Acorn内部会用到它们来影响其它的nodes. 所以为了防止不小心删除了这俩个属性, Espree使用了Object.defineProperty()方法把它们定义成不可枚举的. 如下图 :


通过把这些属性定义为不可枚举的, 任何用户在使用for in遍历 相应的JSON对象时就不会不小心修改了这俩个属性. 但不幸的是, 就是这个做法导致了性能损耗.当我把这俩行注释掉的时候, 性能测试结果就立刻不同了:


就注释了那俩行, esprimaFinishNode()就立刻不再是执行时间最长的函数了, 而且占总时间5%都不到. 而这一切改变都源于Object.defineProperty()俩次调用.


深入理解

现在问题是, 我的确需要 start 与 end 这俩个属性是不可枚举的, 所以我在esprimaFinishNode()中尝试了一些替代方案, 而不是使用Object.defineProperty().

我做的第一件事是 : 使用 Object.defineProperties()来一次定义俩个属性, 而不是分开定义.我的想法很简单, 每一次调用 Object.defineProperty()都会带来很高的性能损耗, 那么为什么不一起定义呢. (译者注: 这个思想跟对HTML Dom 的修改是一个道理). 但很尴尬, 这个思想对于这个问题并无卵用. 所以我推测不是Object.defineProperty()调用次数的原因, 而是只要它被调用了就带来这么多的损耗. (再次调用只对性能有小小的影响)

我想起来当初我读V8优化得到的一些启示了, 我猜原因可能是 : 这个object在定义后又被修改了shape .

Perhaps change the attributes of properties was enough to change the shape of the object in V8, and that was causing a less optimal path to be taken. I decided to this theory.

下一步，　我在想＂创建变量然后使它可枚举＂是否会比＂直接用Object.defineProperty()定义俩个可枚举的变量＂要慢？因此，我又做了一次尝试：


// Faster: ~2650ms 
function Node() {     
    Object.defineProperties(this, {         
         start: { enumerable: false, value: pos, writable: true, configurable: true },        
         end: { enumerable: false, value: pos, writable: true, configurable: true }     
    }); 
}

This version brought the ESLint performance test down even further, to around 2650ms. The easiest way to get it back down to 2500ms? Just make the properties enumerable:

这一次只需要２６５０ｍｓ就完成了．还有木有更快的方案了？那就让这俩个变量可枚举吧．（如下图）（这里ＮＣＺ开个玩笑，意思是我想不到还有其它更优化的方案了，　除非你允许我把这俩个变量变成可枚举的，　那也不需要本文的测试了）

// Fastest: ~2500ms function Node() {     this.start = 0;     this.end = 0; }

所以，　尽量不要使用　＂Object.defineProperty()＂在我看来是最好的方案．
